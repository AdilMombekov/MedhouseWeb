# Журнал разработки — Medhouse / Swiss Energy

Единый лог: что делалось с начала проекта, какая логика использовалась. При каждом обновлении проекта этот файл дополняется новой записью.

---

## Контекст и цели проекта

- **Заказчик:** две компании — **Медхаус** (дистрибьютор парафармации, 15 городов) и **Свисс Энерджи** (контролирующая компания Swiss Group).
- **Цель:** сайт-дашборд для управленческой отчётности: аналитика по имеющимся Excel/данным, импорт отчётов из 1С по шаблонам, роли (админ, модератор, оператор), опционально — подключение Google Drive.
- **Источники данных:** папка **Анализ 2** (годовые файлы 2020–2025, логистика), KATO (регионы), прочие файлы в корне как референсы.

---

## Принятая логика и архитектура

- **Бэкенд:** FastAPI, SQLite (с возможностью перехода на PostgreSQL), JWT-авторизация. Роутеры разнесены по зонам: auth, users, companies, templates, uploads, analytics, dashboard, google_drive.
- **Фронтенд:** React + Vite + Tailwind, SPA с защищёнными маршрутами. Один источник правды по API — вызовы через контекст `AuthContext` с токеном.
- **Данные аналитики:** чтение Excel/CSV с диска по путям из конфига (`Анализ 2`, KATO). При отсутствии файлов или ошибках — демо-данные, чтобы интерфейс не был пустым.
- **Шаблоны выгрузки:** для каждой страницы дашборда — отдельный Excel-шаблон в `backend/templates/`, запись в БД с `report_type` для скачивания по типу.
- **Производительность:** один общий запрос для главной (bootstrap), кэш ответов на бэкенде (5 мин), прогрев кэша при старте сервера.

## Хронология изменений

### 1. Старт проекта — дашборд и бэкенд

- **Backend:** FastAPI, модели User (роли admin/moderator/operator), Company, ReportTemplate, ReportUpload, AnalysisDataset. Авторизация: JWT, bcrypt для паролей. Эндпоинты: `/api/auth/login`, `/api/auth/me`, `/api/users` (для админа), `/api/companies`, `/api/templates`, `/api/uploads`, `/api/analytics/sources`, `/api/analytics/aggregate`, `/api/analytics/preview`.
- **Frontend:** React, роуты Login и под Layout — главная (дашборд), Импорт, Шаблоны, Google Drive, Администрирование. Главная страница: блок «Анализ 2», график по годам, выбор источника и превью таблицы.
- **Логика:** данные для графика и списка источников берутся из папки «Анализ 2»; путь к ней задаётся относительно корня проекта (config: `PROJECT_ROOT`, `ANALIZ2_DIR`).

### 2. Шаблоны и импорт

- Добавлены шаблон для отчётности 1С (создаётся при `init_db` в `backend/templates/`), эндпоинт скачивания по `template_id`. Импорт отчётов: загрузка XLSX/XLS/CSV с указанием типа, компании, периода (доступно модераторам и админам).
- **Логика:** операторы заполняют отчёты в 1С по шаблону и загружают через раздел «Импорт отчётов».

### 3. Google Drive и документация

- Интеграция Google Drive API (OAuth): эндпоинты статуса, списка файлов по ID папки, чтения файла. Документация: README, GOOGLE_DRIVE.md (как получить `credentials.json`, куда положить, первый вход). Учёт проект **pharmacy-bot-487900** (Google Cloud).
- **Логика:** приложение не имеет доступа к аккаунту пользователя; доступ к Drive даёт только настройка OAuth на стороне сервера и один раз пройденная авторизация в браузере.

### 4. Исправления запуска и входа

- Замена passlib на прямое использование **bcrypt** (совместимость версий). Добавлена зависимость **email-validator** для Pydantic EmailStr. Создание БД и первого админа через `scripts/init_db` (admin@medhouse.kz / admin123).
- **Логика:** минимальный набор зависимостей без конфликтов; один скрипт инициализации для воспроизводимого старта.

### 5. Порт фронтенда 3030

- В Vite задан порт 3030 (порты 3000–3020 заняты). Обновлены README и start.ps1.
- **Логика:** избежание конфликтов с другими приложениями на той же машине.

### 6. Демо-данные и исправление «нет данных»

- На бэкенде: при отсутствии файлов или пустом результате отдаются демо-источники и демо-агрегат по годам. Улучшено чтение Excel (пробуем с заголовком и без, суммируем числовые колонки).
- На фронте: исправлена опечатка в Dashboard (источники данных), добавлено отображение «Всего периодов», тултип с числом записей.
- **Логика:** дашборд всегда что-то показывает; при появлении реальных файлов в «Анализ 2» данные подменяются на реальные.

### 7. Семь страниц дашборда и шаблоны выгрузки

- **Страницы:** Сводка (KPI), Продажи, Регионы и филиалы, Логистика, Товарная номенклатура, План–факт, Отчёт для руководства. Для каждой — свой API в `dashboard_router` и данные из «Анализ 2» / KATO (при отсутствии — демо).
- **Шаблоны выгрузки:** 7 xlsx-файлов в `backend/templates/` (dashboard_summary, dashboard_sales, dashboard_regions, dashboard_logistics, dashboard_products, dashboard_plan_fact, dashboard_executive), создаются скриптом `scripts/create_dashboard_templates.py`. Скачивание по типу: `/api/templates/download-by-type/{report_type}`.
- **Логика:** одна страница — один тип отчёта и один шаблон; сотрудники и руководители получают и аналитику, и форму выгрузки в одном месте.

### 8. Деплой и сохранение в Git

- Добавлены `.gitignore`, `backend/uploads/.gitkeep`, в Vite — `base: './'` и настройки build. Создан DEPLOY.md (сборка фронта и бэкенда, Nginx, переменные окружения). Выполнены `npm run build`, `git init`, первый коммит.
- **Логика:** проект готов к выкладке на сервер и к версионированию; инструкции по деплою собраны в одном месте.

### 9. Бесконечная загрузка — обработка ошибок и таймаут

- На фронте: для главной добавлены `.catch`/`.finally` по запросам, таймаут 15 c и AbortController, сообщение об ошибке при недоступности бэкенда. Гарантированное снятие состояния загрузки при любой ошибке или таймауте.
- **Логика:** сбой сети или бэкенда не должен оставлять интерфейс в «вечной загрузке».

### 10. Оптимизация загрузки данных

- **Один запрос для главной:** введён `/api/analytics/bootstrap`, возвращающий за один вызов и `sources`, и `aggregate`. Главная страница переведена на один запрос вместо двух.
- **Кэш на бэкенде:** модуль `app/cache.py` (TTL 5 мин). Кэшируются: bootstrap, sources, aggregate, preview (analytics); summary, sales, regions, logistics, products, plan_fact, executive (dashboard). Повторные запросы отдаются из памяти без чтения Excel/CSV.
- **Прогрев кэша:** при старте приложения в фоновом потоке вызывается расчёт bootstrap и результат кладётся в кэш, чтобы первый заход пользователя был быстрым.
- **Фронт:** таймаут для главной уменьшен до 8 секунд.
- **Логика:** снижение нагрузки на диск и количества запросов; «всё грузится сразу» за счёт одного bootstrap-запроса и кэша на 5 минут.

### 11. Диагностика «Загрузка данных...» и TROUBLESHOOTING

- Добавлен **TROUBLESHOOTING.md**: чек-лист диагностики, когда дашборд висит на «Загрузка данных...». Адаптирован под проект: ключ токена **`medhouse_token`**, один запрос **`/api/analytics/bootstrap`**, порты 3030/8000, Vite proxy, проверка БД и логов uvicorn. В README добавлена ссылка на этот файл.
- В **vite.config.js** для прокси `/api` добавлен `secure: false`.
- **Логика:** единая инструкция по поиску причин рассинхрона frontend/backend; при следующих инцидентах обновлять TROUBLESHOOTING.md под новые эндпоинты или шаги.

### 12. API-ключ для сторонних программ

- В конфиг добавлена переменная **MEDHOUSE_API_KEY**; при её задании запросы с заголовком **X-API-Key** или **Authorization: Bearer &lt;ключ&gt;** считаются авторизованными с правами admin. В `auth.py`: проверка API-ключа до JWT; при совпадении возвращается объект API_KEY_USER (admin). Эндпоинт **GET /api/auth/api-key-info** (без авторизации) описывает, как подключаться по ключу.
- Добавлены **API_ACCESS.md** (инструкция по Postman/curl), скрипт **scripts/generate_api_key.py** для генерации ключа, в .env.example — MEDHOUSE_API_KEY.
- **Логика:** один ключ в .env даёт полный доступ к API для тестов и интеграций без входа через браузер; подключение сводится к одному заголовку.

### 13. Деплой прод: Supabase, Railway, Vercel и логирование

- **Supabase:** в конфиг добавлены `SUPABASE_URL`, `SUPABASE_ANON_KEY`; для БД используется Postgres connection string в `DATABASE_URL` (Supabase → Settings → Database). В `requirements.txt` добавлен `psycopg2-binary`, в `database.py` — `pool_pre_ping=True`. В `.env.example` и документации описан порядок подключения.
- **Фронт в проде:** введён `VITE_API_BASE`: при сборке на Vercel задаётся URL бэкенда (Railway), все запросы к API идут на него. Модуль `frontend/src/config.js`, использование в `AuthContext` и в вызовах `window.open` для шаблонов.
- **Railway:** в `backend/` добавлены `railway.json` (startCommand с `$PORT`) и `Procfile`. В DEPLOY.md — раздел «Деплой прод: Supabase + Railway + Vercel» с чеклистом.
- **Vercel:** в `frontend/vercel.json` заданы buildCommand, outputDirectory, SPA rewrites. В настройках проекта задаётся Root = `frontend` и переменная `VITE_API_BASE`.
- **Логирование:** модуль `app/logging_config.py` (уровень из `LOG_LEVEL`, опционально вывод в файл `LOG_FILE`). Middleware логирует каждый запрос: метод, путь, статус, время, IP. Логи добавлены в роутеры: auth (успех/неуспех входа), uploads (загрузка, отказ), templates (скачивание, 404), users (создание, дубликат email), analytics (ошибки превью), dashboard (демо-данные), google_drive (502/503, ошибки creds), google_sheets (404). В `.env.example` и ЧТО_ТРЕБУЕТСЯ_БЭК_БД_БЭКАП — переменные LOG_LEVEL, LOG_FILE.
- **CORS:** в конфиг добавлена переменная `CORS_ORIGINS` (список доменов через запятую). В проде на Railway задают URL фронта на Vercel, чтобы браузер не блокировал запросы; по умолчанию `*` для разработки.
- **Логика:** прод на облачных сервисах без своего VPS; единый лог для отладки и аудита; 1С пока только импорт, автообновление не делаем.

## Как обновлять этот файл

При каждом значимом обновлении проекта:

1. В раздел **«Хронология изменений»** добавить новый подпункт с номером и кратким названием.
2. Описать: что сделано (бэкенд/фронт/скрипты/документация), какие файлы или модули затронуты.
3. В одном предложении указать **логику/причину** решения (почему так сделано).

Формат блока:

```markdown
### N. Название обновления

- Что сделано: ...
- Логика: ...
```

Файл для ведения: **DEV_LOG.md** в корне проекта.
