## 1. Анализ ошибок и проблем

### 1.1 Что было в логах и в DEPLOY_LOG (из прошлых деплоев)

| Проблема | Причина | Где проявляется |
|----------|---------|------------------|
| **Healthcheck failed / Connection refused** | Бэкенд не стартует или не слушает нужный порт. | Railway: контейнер падает или порт не $PORT. |
| **DATABASE_URL с портом 5432 (Supabase Direct)** | Прямое подключение к Supabase из Railway часто недоступно (сеть/ограничения). | Нужен **pooler** (порт 6543), хост `*.pooler.supabase.com`. |
| **Tenant or user not found** | Неверный хост pooler (aws-0 vs aws-1 в зависимости от региона/проекта Supabase). | Берять точный URI из Supabase Dashboard → Connection string → Transaction pooler. |
| **Network is unreachable (db.xxx.supabase.co)** | Хост Supabase Direct резолвится в IPv6; Railway не поддерживает IPv6. | Не использовать Direct; только pooler. |
| **sqlalchemy.exc.NoSuchModuleError: sqlalchemy.dialects:https** | В `DATABASE_URL` попала строка `https://...` (например URL проекта Supabase вместо connection string). | Variables в Railway: не путать SUPABASE_URL и DATABASE_URL. |
| **CORS блокирует запросы с фронта** | В `CORS_ORIGINS` указан домен без `https://` или лишний слэш; бэкенд отклоняет preflight. | Задать точный origin, например `https://medhouse-web.vercel.app` (без слеша в конце). |
| **Фронт на Vercel не доходит до API** | Фронт ходит на `/api` (same origin) — на Vercel бэкенда нет. | Нужен базовый URL API (например через переменную окружения), чтобы запросы шли на Railway. |
| **Vercel: билд из корня / неверный preset** | Root Directory не `frontend` или Framework не Vite. | Vercel → Settings → Root Directory = `frontend`, Framework = Vite. |
| **Postgres: relation "pg_stat_statements" does not exist** | Railway Data UI запрашивает расширение, которого нет в образе. | Не влияет на приложение; можно игнорировать или включить расширение в БД по желанию. |

### 1.2 Текущее состояние кода (после отката)

- **backend/app/config.py:** только `DATABASE_URL` из env, по умолчанию `sqlite:///./medhouse.db`. Нет переменных Supabase, CORS, логов.
- **backend/app/database.py:** `create_engine(DATABASE_URL)` без `pool_pre_ping`; для SQLite — `check_same_thread=False`. Нет драйвера Postgres в коде (драйвер задаётся строкой подключения или по умолчанию SQLAlchemy).
- **backend/requirements.txt:** нет `psycopg2-binary` — для PostgreSQL на Railway его нужно добавить.
- **backend/main.py:** только `GET /health` (без проверки БД). CORS `allow_origins=["*"]`.
- **Фронт:** в `AuthContext.jsx` константа `API = '/api'` — все запросы на относительный путь. Локально работает за счёт proxy в `vite.config.js` (`/api` → `http://127.0.0.1:8000`). На Vercel домен один (фронт), бэкенда нет — запросы на `/api` уйдут на Vercel и будут 404. Нужен механизм подстановки базового URL API в прод (например env-переменная и использование её в одном месте).
- **Скачивание шаблонов:** в коде есть `window.open('/api/templates/...')` — тоже относительный путь; в проде должен учитывать базовый URL бэкенда.
- **Нет файлов:** `backend/railway.json`, `backend/Procfile`, `backend/runtime.txt`, `frontend/vercel.json` — их не было в коммите 5093188; для деплоя на Railway и Vercel их нужно будет добавить по плану.

## 2. Тесты и ревью

### 2.1 Тесты

- В репозитории **нет автоматических тестов** (нет `test_*.py`, pytest и т.п.).
- Рекомендуется перед деплоем вручную проверить: локальный запуск бэка и фронта, вход, дашборд, один запрос к API (например `/api/analytics/sources` или `/api/auth/me`), скачивание шаблона. После деплоя — то же с прод URL.

### 2.2 Ревью по вертикали (стек по слоям)

| Слой | Что проверить |
|------|----------------|
| **БД** | Сейчас только SQLite. Для Railway: источник данных — либо Railway Postgres (рекомендовалось), либо внешний (например Supabase). Нужны: корректный `DATABASE_URL`, драйвер Postgres в requirements, при необходимости `pool_pre_ping` для стабильности соединений. |
| **Бэкенд** | FastAPI, uvicorn. Для Railway: слушать `0.0.0.0` и порт из `$PORT`; один раз инициализировать таблицы и данные (init_db, при необходимости шаблоны). Health: минимум `/health`; полезно иметь `/health/ready` с проверкой БД для оркестратора. |
| **API** | Роутеры под `/api`. CORS: в проде лучше ограничить `allow_origins` списком доменов фронта (Vercel), а не `*`. |
| **Фронт** | Сборка: Vite, `npm run build`, выход в `dist/`. В проде все вызовы API должны идти на URL бэкенда (Railway), а не на `/api` текущего домена. |
| **Деплой** | Railway: сборка из папки `backend`, переменные окружения, привязка БД. Vercel: сборка из `frontend`, переменная для базового URL API. |

### 2.3 Ревью по горизонтали (потоки данных)

- **Логин:** фронт → POST `/api/auth/login` → бэкенд → БД (users). В проде: фронт на Vercel должен слать запрос на `https://<railway-domain>/api/auth/login`.
- **Токен и запросы:** после логина запросы с заголовком `Authorization: Bearer <token>`. Все вызовы `api(path)` и прямые `fetch` должны использовать один базовый URL.
- **Скачивание файлов:** `window.open(API_BASE + '/api/templates/...')` — в проде API_BASE = URL Railway.
- **Данные аналитики:** чтение с диска (папка «Анализ 2») и/или кэш; на Railway диск эфемерный — либо только демо/кэш, либо внешнее хранилище (не в рамках текущего отката).

## 3. Зачем какие системы и почему их выбирали

| Система | Назначение | Зачем выбирали |
|---------|------------|----------------|
| **Railway** | Хостинг бэкенда (FastAPI) и при необходимости БД (Postgres). | Один сервис для бэка и БД, простой деплой из Git, автоматическая подстановка `DATABASE_URL` при добавлении Postgres. Не нужен свой VPS. |
| **Vercel** | Хостинг фронтенда (статическая сборка React/Vite). | Деплой из того же репо, автоматические билды и превью, CDN. Хорошо сочетается с отдельным API на Railway. |
| **Supabase** | Вариант внешней БД (PostgreSQL) и при необходимости клиентские фичи (Auth, Realtime). | Если не использовать Railway Postgres — можно взять БД в Supabase и подключаться по connection string (обязательно через pooler, порт 6543). В текущем откате в коде Supabase нет; добавлять только по твоему запросу. |

Изначально план был: БД (Supabase или Railway Postgres) + бэкенд на Railway + фронт на Vercel. После отката привязка «всех систем к Railway» означает: Railway как единственная платформа для бэкенда и БД (Postgres), Vercel только для фронта; Supabase — только если ты решишь использовать его как БД вместо Railway Postgres.

## 4. План работы по деплою и привязке к Railway

Ниже — план без внесения изменений в код. Каждый шаг можно выполнять по твоей команде.

### 4.1 Railway: подготовка репозитория и бэкенда

1. **Сборка из папки backend**  
   В настройках сервиса Railway указать **Root Directory** = `backend`, чтобы сборка (Nixpacks/пакетник) видел Python и `requirements.txt`.

2. **Запуск приложения**  
   Добавить в репозиторий способ запуска с портом из окружения (например `railway.json` или `Procfile`):  
   `uvicorn app.main:app --host 0.0.0.0 --port $PORT`.  
   При необходимости зафиксировать версию Python (например `runtime.txt`).

3. **БД**  
   - Вариант A: в проекте Railway добавить сервис **Postgres**, привязать к сервису бэкенда переменной **Reference** `DATABASE_URL`.  
   - Вариант B: внешняя БД (например Supabase) — задать `DATABASE_URL` вручную (обязательно pooler, порт 6543).  
   В обоих случаях в `requirements.txt` должен быть драйвер Postgres (`psycopg2-binary`), если используется PostgreSQL.

4. **Переменные окружения**  
   Минимум: `SECRET_KEY`, при необходимости `CORS_ORIGINS` (список разрешённых origin через запятую, например URL Vercel). Не смешивать в одной переменной URL проекта (типа Supabase URL) и connection string БД.

5. **Инициализация БД**  
   Один раз выполнить создание таблиц и начальных данных (аналог `python -m scripts.init_db` и при необходимости шаблоны) — через Run Command / one-off в Railway или локально с `DATABASE_URL` от Railway.

6. **Домен**  
   В Railway выдать домен сервису бэкенда и использовать его как базовый URL API для фронта.

### 4.2 Vercel: фронтенд

1. **Проект**  
   Импорт того же репозитория, **Root Directory** = `frontend`, **Framework** = Vite, сборка и выход по умолчанию (например `npm run build`, `dist`).

2. **Базовый URL API**  
   В коде фронта заменить жёстко заданный `/api` на значение из переменной окружения (например `VITE_API_BASE`), чтобы в проде запросы шли на домен Railway. Настроить в Vercel переменную `VITE_API_BASE` = URL бэкенда Railway (без слеша в конце).

3. **Скачивание шаблонов**  
   Все вызовы вида `window.open('/api/...')` заменить на использование того же базового URL API, чтобы скачивание работало с прод-бэкендом.

4. **Деплой**  
   После деплоя взять URL фронта и прописать его в `CORS_ORIGINS` на Railway (если в бэкенде будет использоваться список origin).

### 4.3 Проверка

1. Открыть `https://<railway-domain>/health` — ответ 200.  
2. При наличии эндпоинта проверки БД (например `/health/ready`) — убедиться, что БД в статусе connected.  
3. Открыть фронт на Vercel, войти, проверить дашборд и один раздел, скачивание шаблона.
