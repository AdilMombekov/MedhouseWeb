# Архитектура продукта и техническое задание

Полное ТЗ на систему управленческой отчётности Медхаус / Свисс Энерджи: архитектура прода, связи данных, логирование, ревью, безопасность, масштаб и требования к инфраструктуре. Дополнения к вашим идеям отмечены отдельно; там, где вы указали «не менять» — форму и логику не меняем.

---

## 1. Масштаб и пользователи

| Параметр | Значение |
|----------|----------|
| Активных пользователей (работа в системе) | **100–150 человек** |
| Просматривающих (доступ на просмотр) | **до ~300 человек** |
| Организации | Медхаус, Свисс Энерджи (и объединённый свод) |
| Регионы/города | 15+ (Медхаус), филиалы Свисс |

Итого: проектирование на **~300 одновременных просмотров** и **до 150 активных сессий** (запросы к API, загрузки, редактирование).

## 2. Архитектура продукта (прод)

### 2.1 Общая схема

```
[Пользователи: браузер, мобильное приложение]
         │
         ▼
[Балансировщик / Reverse Proxy (Nginx / cloud LB)]
         │
    ┌────┴────┐
    ▼         ▼
[Frontend]  [Backend API]
  (SPA)      (FastAPI)
    │         │
    │    ┌────┼────┬─────────────┬──────────────┐
    │    ▼    ▼    ▼             ▼              ▼
    │  [БД] [Кэш] [Файлы]   [1С выгрузки]  [Google API]
    │
    └── Статика (CDN опционально)
```

- **Frontend:** SPA (React/Vite), сборка в статику; раздача через Nginx или CDN.
- **Backend:** FastAPI; аутентификация (JWT + опционально 2FA для админов); роли и иерархия доступа.
- **БД:** основная рабочая БД (PostgreSQL рекомендуется при 100+ пользователях; SQLite допустим для старта с последующей миграцией).
- **Кэш:** Redis (сессии, кэш отчётов, очереди задач) при росте нагрузки.
- **Файлы:** загруженные Excel, выгрузки, бэкапы — отдельное хранилище (диск сервера или S3-совместимое облако).
- **Внешние данные:** 1С (выгрузки/файлы, в перспективе API), Google (Drive, Sheets, Calendar, Meet), Kaspi/Halyk API при наличии.

### 2.2 Связи данных (откуда что берётся)

- Описано в **[ИСТОЧНИКИ_ДАННЫХ_И_ЛОГИКА_РАСЧЕТОВ.md](ИСТОЧНИКИ_ДАННЫХ_И_ЛОГИКА_РАСЧЕТОВ.md)** по блокам (ПНЛ, мотивация, касса/банк, остатки, карта и т.д.).
- Общая логика:
  - **Факт:** 1С (выгрузки Excel/CSV или будущий API), загрузка через сайт или автоматическая выгрузка в папку/Google.
  - **Планы:** формат бюджета (папки Евгений/Алмас), свод в Google Drive; заливка на сайт с сохранением формата и формул.
  - **Справочники:** организации, города, номенклатура, контрагенты — БД сайта (наполнение из 1С или вручную).
  - **Карта:** координаты/адреса из загружаемых данных + бот (остатки по аптекам) + данные сетей.

### 2.3 Логирование (действия пользователей)

- **Что логировать:** вход/выход, открытие разделов, загрузка файлов, сохранение правок (ПНЛ, мотивация, планы, заявки на оплату, комментарии к дебиторке и т.д.), согласования (бухгалтер, фин. директор), экспорт отчётов.
- **Формат:** таблица (или отдельная БД) логов: `user_id`, `role`, `action`, `entity_type`, `entity_id`, `old_value` (опционально), `new_value` (опционально), `ip`, `timestamp`.
- **Хранение:** логи не удаляются обычными пользователями; срок хранения по политике (например, 2–3 года).

### 2.4 Бэкапы и ревью

- **Бэкап каждые 3 часа:** полный снимок БД + файлы загрузок и, при необходимости, снимок логов действий (чтобы администратор мог восстановить «кто что делал»).
- **Ревью по вертикали:** иерархия доступа (город видит свой прогресс, руководитель региона — свой регион, управленцы — всё); согласование оплат: сотрудник → бухгалтер → фин. директор.
- **Ревью по горизонтали:** при необходимости — кросс-проверка между отделами (например, продажи vs логистика по остаткам); отчёты и дашборды дают единые цифры из одной БД.

## 3. Требования по функционалу (краткая спецификация)

Ниже — тезисы по разделам; детали источников и расчётов — в документе по источникам данных.

- **ПНЛ (уровень 1):** отдельно Медхаус / Свисс; факт из 1С (ПНЛ 2026 и др.); планы из формата бюджета (Евгений/Алмас), свод в Google; периоды месяц/квартал/полугодие/год; ручной курс; ручные правки с версионированием; два варианта при правке формул (первоначальный/исправленный); отчёт по одной организации или объединённый.
- **Мотивация (MX):** реал 8 мес, дзад, остаток, прайс; реализация за 8 мес; новый план с предупреждением при превышении остатка; ручное распределение остатков по городам; обновление планов до 10 числа; умная загрузка (новые — в учёт, существующие — сообщение «уже есть» и показ совпадений); загрузка через Excel или Google API.
- **Касса, Банк, Акниет:** касса и банк из 1С по городам/счетам; KPI-планирование; ежедневная реализация (детализация по контрагентам/номенклатуре/складам — позже).
- **KPI и расписки:** расчёт KPI по торговым; расписки по работникам по городам (иерархия); управленцы видят все планы.
- **Результаты продаж:** планы по городам/номенклатуре/брендам; факт и новый план; контроль в динамике по графикам (прогресс по дням).
- **Остатки и карта:** загрузка по контрагентам/номенклатуре/документам/суммам по дням; скользящее 12 мес; средние остатки по контрагентам; карта с слоями (деньги, регионы, ТП, бренды); бот — остатки по аптекам; сети — их остатки.
- **ABC, склады, ДС, дебиторка:** ABC по контрагентам/номенклатуре/регионам; остатки по складам по городам; эффективность логистики; ежедневный остаток ДС; оборот за период; дебиторка с комментариями ответственных; прогнозы.
- **ОС, бухгалтерия:** ОС и амортизация из 1С; расчётные листки; расходы по документам 1С; заявки на оплату с прикреплёнными счетами; согласование бухгалтер → фин. директор; денежные дни вт/чт до 16:00; выгрузка по платежам.
- **Логи и бэкапы:** логирование всех значимых действий; бэкап каждые 3 часа; отдельный лог ошибок.
- **Meet, календарь, мессенджер:** интеграция Google Meet; календарь; мессенджер и группы.
- **Розница:** план-факт, ABC; Kaspi/Halyk API при наличии.
- **Маркетинг:** спрос, анализ респондентов; утверждение плана затрат на маркетинг (месяц/неделя).
- **Зарплаты:** контроль; фото-история расчётных листов в бэкапе (по году/месяцу или по сотруднику).
- **Объявления и приложение:** всеобщие объявления; кроссплатформенное мобильное приложение (Android, iOS).
- **Бюджет:** формат бюджета на сайте; выгрузка за год с сохранением формул; новая логика учёта и импорт.
- **Себестоимость и склад:** складские, логистика, перемещения; остатки по складам по городам; ТНВЭД/НКТ; лоты; лицензии; срок годности; ожидание завоза; расчёт себестоимости.
- **Налоги:** контроль налогов; папка бухгалтерия на D:; ТОО нормы; автообновление воскресеньями с сайта новостей НК.
- **Юристы:** шаблоны договоров; реестр договоров; реестр судебных дел и писем.
- **Сервис:** % брака; запас запчастей; колл-центр; обратная связь по качеству.
- **Видеокамеры:** вкладка с последующей доработкой.
- **ТП и маршруты:** ежедневная реализация по контрагентам → работа ТП (посещения аптек); планирование маршрутов супервайзерами.

## 4. Системы для полноценной работы

### 4.1 База данных

- **Сейчас:** SQLite допустим для прототипа и малой нагрузки.
- **Для прод при 100–150 пользователях:** **PostgreSQL** (надёжность, параллельные запросы, бэкапы, репликация).
- Объём: ориентировочно **10–50 ГБ** на старте (отчёты, логи, версии правок, файлы метаданных); рост за счёт логов и загрузок — предусмотреть увеличение диска.

### 4.2 Хостинг и серверы

- **Вариант 1:** VPS/облачный сервер (например, 4 CPU, 8–16 ГБ RAM, SSD 100+ ГБ); на нём — Nginx, backend, БД, скрипты бэкапов.
- **Вариант 2:** разделение: **application server** (backend + frontend) + **DB server** (PostgreSQL) + **file storage** (диск или S3).
- Рекомендация: начать с одного сервера; при росте — вынести БД и хранилище файлов.

### 4.3 Облачное хранилище и бэкапы

- **Всегда доступные (горячие) данные:** текущая БД, последние загрузки, кэш — на основном сервере или в облаке с быстрым доступом.
- **Бэкапы:** полные снимки БД и критичных файлов каждые 3 часа; копии в **отдельное хранилище** (другой диск/облако, например S3 или аналог). Хранение: минимум 7–30 дней суточных, 3–12 месячных.
- Ориентир по объёму облака: **50–200 ГБ** под бэкапы и файлы (с ростом — масштабировать).

### 4.4 Подписки и внешние сервисы

- **Google Workspace / API:** Drive, Sheets, Calendar, Meet — по числу активных пользователей и лимитам API.
- **1С:** лицензии и выгрузки как уже есть; в перспективе — HTTP API 1С при наличии.
- **Почта/SMS:** для уведомлений и восстановления доступа (опционально).
- **Мобильное приложение:** сборка под Android/iOS (React Native или Flutter) — учёт в подписках магазинов (если публикуете от своего аккаунта).

### 4.5 Шаблонирование и клонирование под другие компании

- Вынести в конфиг: названия организаций, список ролей, иерархия (регионы/филиалы), перечень модулей (ПНЛ, мотивация, касса и т.д.).
- Данные и справочники — отдельно по «тенанту» (одна компания или группа компаний); при мультитенантности — разграничение по `company_id` / `tenant_id`.
- Документация по развёртыванию: скрипты миграций БД, переменные окружения, шаги установки — чтобы можно было развернуть копию под другую организацию.

## 5. Защита данных и конфиденциальность

### 5.1 Общие принципы

- Доступ только по аутентификации (логин/пароль; опционально 2FA для админов и фин. директора).
- Разграничение по ролям и иерархии (город видит своё, управленцы — свод).
- Передача по HTTPS; пароли и секреты не в коде, только в переменных окружения/секретах.

### 5.2 Администрирование и права

- **Только вы (администратор)** выдаёте права, регистрируете пользователей.
- Регистрация по **email** и при необходимости по **номеру телефона** (верификация по коду/SMS или только email).
- Роли задаются вручную (например: администратор, фин. директор, бухгалтер, руководитель региона, торговый, городской менеджер, только просмотр).
- Возможность **бана** пользователя (блокировка входа без удаления истории).

### 5.3 Персональные данные и анкетирование

- Регистрация **личных данных** и **анкетирование** пользователей (хранить в БД с соблюдением политики конфиденциальности).
- **Аватарка** пользователя (загрузка фото, хранение в защищённом каталоге или облаке с доступом только для авторизованных).

### 5.4 Контроль работы пользователей

- **Логирование действий** (см. выше) — кто что открыл, изменил, выгрузил.
- **Возможность просмотра администратором** любой вкладки/раздела от имени любого пользователя (режим «как пользователь» для проверки прав и жалоб) — только для вашей роли, без разглашения этой возможности остальным.

### 5.5 Ограничение копирования и экспорта

- **Никто, кроме вас, не должен знать полную логику работы** (формулы, алгоритмы, связи) — на бэкенде логика не отдаётся клиенту; фронт только отображает данные и формы.
- **Запрет или ограничение копирования логики прода:** код бэкенда и скрипты не доступны пользователям; минификация фронта; критичные расчёты только на сервере.
- **Скриншоты:** по вашему требованию — запрет скриншотов на сайте/в приложении (например, `user-select: none`, запрет копирования буфера, предупреждение в правилах). Ограниченно: разрешить скриншот только определённой области (например, только графики без таблиц с цифрами) — технически сложно на 100%, но можно снизить удобство копирования (водяные знаки, отключение копирования текста).
- **Экспорт данных:** только с **явного разрешения** (роль «может экспортировать» или разовая выдача права администратором); логирование каждого экспорта (кто, когда, какой отчёт/период).

### 5.6 Интеллектуальная защита

- Условия использования и NDA при регистрации (текст о неразглашении данных и логики).
- Соглашение о конфиденциальности; в логах фиксировать факт экспорта и доступа к чувствительным разделам.
- Дополнительно (по желанию): водяные знаки на печатных/экспортных формах с датой и пользователем для прослеживания утечек.

## 6. Дополнения (без изменения заданной вами логики и формы)

- **Микроблогинг:** внутренняя лента коротких сообщений/обновлений по отделам или по проектам — для быстрого обмена новостями без перегрузки мессенджера.
- **Единый справочник контрагентов и номенклатуры:** один источник правды из 1С или с сайта, чтобы все отчёты (ПНЛ, мотивация, остатки, карта) использовали одни и те же коды и названия.
- **Уведомления:** напоминания о сроках (план до 10 числа, денежные дни вт/чт, согласование заявок) — в интерфейсе и при возможности по email/push в приложении.
- **Единый модуль «Заявки»:** не только оплаты, но и заявки на отпуск, командировки и т.д. с маршрутом согласования — по мере необходимости.
- **Дашборд для фин. директора:** один экран с ключевыми цифрами (касса, банк, дебиторка, план-факт по ПНЛ, статус согласований) без изменения существующей логики расчётов.
