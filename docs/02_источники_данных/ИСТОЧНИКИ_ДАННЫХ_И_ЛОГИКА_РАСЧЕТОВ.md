## 1. Уровень 1 — Руководство (ПНЛ)

### 1.1 Назначение

- ПНЛ (прибыль и убытки / P&L) **отдельно по Медхаус** и **отдельно по Свисс Энерджи**.
- Факт строится из **объединения данных** в Google Drive (по каждому месяцу): в объединение тянутся счета **1010** и **1030** (1С); из них формируется **разница**, которая раскидывается по показателям; итоговая разрезка по статьям идёт из **свода** (Свисс). Все формулы листа «объединение данных» и связь 1010/1030 → разница → свод нужно зафиксировать и воспроизвести на сайте.
- **Планы:** в **левом конце листа января** (или первого месяца года) вставляются данные **БДР формата бюджета**; на их основе строится план на год. Источник БДР — папки в Google Drive, в названии которых **в конце** указано **Евгений** (Медхаус) или **Алмас** (Свисс); в этих папках лежит формат бюджета.

### 1.3 Логика расчёта

- **Периоды:** по месяцам (с датой), по кварталам, по полугодиям, по году.
- **Курс:** ручная корректировка курса за каждый месяц (хранить в БД/справочнике).
- **Ручные правки:** с перезаписью в новую версию/БД; при правке ячеек с формулами — два варианта (первоначальный по формуле / исправленный вручную), расчёты не сбиваются.
- **Организации:** отчёт по одной (Медхаус или Свисс) или объединение двух и более.

## 2. Уровень 2 — Мотивация (MX-формат)

**Расположение:** Google Sheets — книга **МХ формат**: [таблица](https://docs.google.com/spreadsheets/d/1G5atWyRUOxr23iZ3Q8txMRo_DrdjdCEo9VVQfVBAl6Q/edit?gid=1011251887#gid=1011251887) (ID: `1G5atWyRUOxr23iZ3Q8txMRo_DrdjdCEo9VVQfVBAl6Q`).

### 2.1 Текущая логика (разбор и проблемы)

- **Реал 8 мес:** лист с данными за 8 месяцев; из него формируется лист **«Продажи дистр»**; из «Продажи дистр» считается **средняя реализация**, от которой управленцы ставят планы.
- **Текущий расчёт планов в «Реал 8 мес»:** берётся сумма **с возвратами** в столбце **G**; через процент от этой суммы получают новую сумму «согласно плану». Это вам не нравится и считается некорректным: реализация за 8 мес ~230 млн, план на 30–40 млн выше; логичнее планировать от тренда/средней, а не от процента к сумме с возвратами.
- **Коэффициент в мотивации:** от плана/факта коэффициент попадает в столбец **J** листа «Реал 8 мес»; в **Q** показывается, что нужно вставить в **R**. После единоразовой вставки в R система ещё раз пересчитывает Q и сообщает «не хватает ещё столько» по коэф. Сейчас вы вручную подбираете два коэф, пока Q не покажет 0 по каждой ячейке.
- **Подгонка к плану:** сначала в «Реал 8 мес» подгоняете реал до суммы, равной плану (столбец **X** в «Продажи дистр»); затем коэффициент раскидывает ровно в мотивации по городам и сотрудникам. Итог — нечёткая постановка KPI и трудоёмкая ручная настройка.
- **План по одной номенклатуре на 5–6 городов:** коммерческий иногда ставит план по столбцу X только на одну номенклатуру и раскидывает на 5–6 основных городов; на остальные города план не кидается. В «Реал» такое считать крайне неудобно, приходится хардкодить. Нужна явная поддержка сценария «план только по выбранным городам/номенклатуре».
- **Контроль цен и остатков:** требуется контроль ценовых данных по номенклатуре и остатков (чтобы план не превышал доступный остаток и соответствовал прайсу).

### 2.2 Источники данных (зафиксировано)

| Данные | Лист/источник | Столбцы/примечание |
|--------|----------------|---------------------|
| Реал за 8 мес | Лист «Реал 8 мес» | G — сумма с возвратами; J — % (коэф из мотивации); Q — что вставить; R — вставленное значение; связь с планом (X в «Продажи дистр»). |
| Продажи дистр | Лист «Продажи дистр» | Формируется из «Реал 8 мес»; средняя реализация; X — план (целевая сумма, до которой подгоняют реал 8 мес). |
| Дзад | Лист «Дзад» | Задолженность по городам; в мотивации слева — факт от дзад и план вручную; по коэф умножается так, чтобы новый факт соответствовал плану (логика плана по дебиторке/возвратам). |
| Остаток, прайс | Отдельные листы/файлы | Контроль остатков и цен по номенклатуре; проверка при установке плана. |

### 2.3 Требования к новой модели KPI и расчёта планов

1. **План от средней реализации, а не от % к сумме с возвратами:** считать среднюю реализацию за 8 мес по городам/номенклатуре (или по компании с разбивкой); план задавать в тг или в % к средней, без «подгонки» суммы в G процентом.
2. **Один явный коэффициент распределения:** вместо подбора двух коэф до нуля в Q — одна формула/алгоритм: по целевому плану (X) и текущему реал 8 мес автоматически считать коэффициент так, чтобы после распределения по городам/сотрудникам суммарный факт соответствовал плану; при необходимости показывать «остаток не хватает на план» и блокировать или предупреждать.
3. **Распределение по городам и сотрудникам:** коэффициент раскидывает план ровно по мотивации (города, сотрудники); при частичном плане (только 5–6 городов или одна номенклатура) — явное указание «по каким строкам/городам план ставится», без хардкода в реал.
4. **Контроль остатков и цен:** при установке плана проверять: план ≤ доступный остаток по позиции; при необходимости проверка по прайсу (объём × цена).
5. **Ревью и тест:** после внедрения новой логики — ревью и тест на исторических данных (реал 8 мес ~230 млн, план 260–270 млн), чтобы убедиться, что распределение и KPI считаются корректно.

### 2.4 Умная загрузка БД

- Новые данные — в учёт; при совпадении с уже имеющимися — сообщение «уже есть» и показ совпадений.
- Загрузка через Excel или Google API; форму MX сохранить, логику не усложнять сверх описанного.

## 4. KPI и расписки по работникам

### 4.1 KPI по торговым

- Расчёт KPI по торговым — на основе планов (мотивация, до 10 числа) и факта (реализация, отгрузки). Связь с блоком мотивации: коэф план/факт → столбец J в «Реал 8 мес», распределение по городам и сотрудникам (см. раздел 2). Новая модель KPI — в п. 2.3.
- **По дебиторке/возвратам:** план считается от листа **«Дзад»** (задолженность по городам). В мотивации слева — **факт от дзад** и **план вручную**; по коэффициенту умножается так, чтобы новый факт соответствовал плану.

### 4.2 Расписка по работникам на новый месяц

- Город видит только свой прогресс; управленцы — все планы. Данные с привязкой к иерархии (город, отдел, сотрудник).

- Связь с выгрузкой отгрузки из 1С (имя отчёта, столбцы).

## 6. Остатки по контрагентам и карта

### 6.1 Новая логика расчёта остатков (Power BI / сайт)

- **Источник реализации:** вся реализация за последние **5 лет** из Excel-таблиц **2021–2025**: контрагент, реализация по дням (или по месяцам).
- **По месяцам:** через **СУММЕСЛИМН** (или аналог) берётся реализация по каждому контрагенту за каждый месяц и последующие месяцы.
- **Средние продажи в месяц:** логика «от обратного»: из реализации **предыдущего месяца** вычесть реализацию **следующего месяца** → разница трактуется как средние продажи в месяц (или средний расход/затрачиваемость).
- **Если разница больше реализации:** брать срез по **2 недели** (две недели и следующие 2 недели), чтобы выяснить, за сколько дней средняя реализация равна средней затрачиваемости. Когда средняя реализация и остаток в модели выходят в 0 — это «средний расход». За 2 недели → умножить на 2 для средних затрат по товарам; далее делить/умножать, чтобы оценить, к какому дню примерно какие остатки у контрагента.
- **Тест и ревью:** проверить на данных, работает ли такая логика (разница мес N и мес N+1 как средние продажи; 2-недельные срезы для затрачиваемости). Если да — применять на сайте.
- **Применение:** СУММЕСЛИМН по **контрагент**, **адрес**, **номенклатура**; по каждой номенклатуре — остаток; по каждому адресу (если сеть или больше 1 аптеки на контрагента) — отдельно. Итоговая таблица остатков по контрагентам/адресам/номенклатуре для карты и диаграмм.

### 6.2 Истинные остатки от ТП (бот)

- **Таблица «Заявка фармацевты»:** при каждом посещении торговый фиксирует остатки на текущий день. Данные попадают в бот; бот размещён в **D: dev bot1 фармаси** (уже задеплоен и работает).
- **Фиксация в источниках:** формат заявки фармацевтов (столбцы, типы данных) описать в этом разделе; на сайте — якорный расчёт по этому формату.
- **Форма для расчёта (сети):** данные от сетей по остаткам пока не передаются; когда будут — применить к той же форме расчёта. Ссылка на форму: [Google Таблица](https://docs.google.com/spreadsheets/d/1XSFq5PbJadJR21h6Z9z3hhfnGu4tEl93K6WEXUYutdI/edit?gid=1144561640#gid=1144561640) (доступ по авторизации; структуру столбцов при возможности вынести в описание).

### 6.3 Два вида представления и карта

- **Два вида таблиц:** 1) расчётные остатки по реализации (п. 6.1); 2) фактические остатки от ТП/бота (п. 6.2). Их можно **объединять** или смотреть **раздельно**.
- **Визуализация:** по **диаграмме** (свод по городам/брендам) и по **карте** — раскидка по адресам и координатам в GIS (слои: деньги, регионы, ТП, бренды).

## 7. ABC, склады, ДС, дебиторка, прогнозы

### 7.1 ABC-анализ

- По контрагентам, номенклатуре, регионам (и при необходимости по каналу, бренду). Источники: данные реализации и остатков (разделы 2, 6).

### 7.2 Остатки по складам (города) — зафиксировано

- В **МХ формат** есть лист **«Оста»**: выгрузка остатков из 1С. По умолчанию выгружается **за целый месяц**, но можно **каждый день обновлять**, чтобы видеть на начало дня остаток по товарам по каждому складу или обороты товара за период. Проверить по вашей базе и зафиксировать в источниках: имя отчёта 1С, столбцы (склад, город, номенклатура, остаток, период).

### 7.3 Дебиторка — зафиксировано

- Берётся из **ОСВ по счёту 1210** (дебиторская задолженность). Готовые отчёты бывают разные: например **«Дзад»** — по городам; можно сформировать и **по контрагенту**. Проверить по 1С и зафиксировать логику в источниках (какой отчёт, какие измерения).

### 13.1 ТП и маршруты — зафиксировано

- **Корректировка супервайзером:** на карте возможность **удобно выделить точки или район** и **автоэкспорт по рутам**, чтобы торговый видел новые руты и на карте отображались его будущие маршруты на месяц.

### 13.2 Логистика — зафиксировано

## Ревью: что понял / что не понял

Ниже — краткое ревью по вашим пояснениям: что зафиксировано как понятное, что требует уточнения или доступа к файлам.

### ПНЛ (уровень 1)

- **Понял:** В Google Drive по каждому месяцу ПНЛ есть лист/блок «объединение данных», куда тянутся счета **1010** (касса) и **1030** (банк) из 1С; из них считается **разница**, которая раскидывается по показателям; итоговая разрезка по статьям идёт из листа **«свод»**. Планы: в левом конце листа **января** вставляются данные **БДР формата бюджета**, оттуда строится план на год.
- **Не понял / нужно:** Точные формулы листа «объединение данных» (какие ячейки ссылаются на 1010/1030, как считается разница и связь с «свод») — без доступа к книге ПНЛ в Google или выгрузки структуры формул воспроизвести нельзя. Нужна выгрузка формул или скрин с именами диапазонов/ячеек.

### Мотивация (MX)

- **Понял:** Реал 8 мес → лист «Продажи дистр» → средняя реализация; планы сейчас через столбец G (сумма с возвратами) и процент — это вы считаете некорректным. Коэф из мотивации попадает в J, в Q показывается что вставить в R; подбор двух коэф до нуля в Q; подгонка реала к плану (X в «Продажи дистр»), затем раскидка по городам/сотрудникам. Дзад — факт и план вручную, по коэф выравнивание к плану. Нужна новая модель: план от средней реализации, один явный коэф, контроль остатков и цен, поддержка частичного плана (5–6 городов, одна номенклатура).
- **Не понял / нужно:** Точные диапазоны листов «Реал 8 мес» и «Продажи дистр» (номера столбцов X, G, J, Q, R и связь с листом мотивации по городам) — для реализации новой логики на сайте лучше иметь копию структуры или экспорт формул.

### Касса, банк

- **Понял:** Касса = 1010 + организация в ОСВ; банк = 1030 + организация в ОСВ. В МХ формат листы «Касса» и «Банк» — ежедневная вставка отчёта из 1С с 1 по текущее число. Зафиксировано.

### Результаты продаж

- **Понял:** Первый лист — вертикаль номенклатура, горизонталь города, по городам по брендам (ВПР и СУММЕСЛИ); дашборд от таблиц «Отгрузка 1» и «Выгрузка 2»; план/факт %; «Лидеры» — сортировка по выполнению; «План по брендам» — СУММЕСЛИ по брендам, факт из отгрузки. Нужно зафиксировать диапазоны и формулы по вашей книге.

### Остатки и карта

- **Понял:** Реализация 2021–2025 по контрагентам по дням; СУММЕСЛИМН по месяцам; разница «предыдущий месяц минус следующий» как средние продажи в месяц; при необходимости срез по 2 недели для затрачиваемости; остаток по контрагент/адрес/номенклатура. Бот в D: dev bot1 фармаси, таблица «Заявка фармацевты» — фиксация остатков ТП. Форма по ссылке — для якорного расчёта; данные сетей пока не передаются. Два вида: расчётные остатки и факт от ТП; объединение или раздельный просмотр; карта по адресам/координатам.
- **Не понял / нужно:** Форму «заявка фармацевтов» по ссылке не удалось открыть (требуется вход в Google). Нужно описание столбцов формы или доступ на просмотр для фиксации формата.

### Остатки по складам, дебиторка

- **Понял:** Лист «Оста» в МХ формат — выгрузка из 1С, можно ежедневно; дебиторка из ОСВ 1210, отчёты типа «Дзад» по городам или по контрагенту. Зафиксировано; при реализации проверить по вашей 1С имена отчётов.

### ОС, бухгалтерия

- **Понял:** Амортизация — отчёт 1С, квартальное обновление. Бухгалтерия — субконто статья ДДС или документ «Движение денежных средств», реестр проведённых платежей, платёжные поручения, расходные ордера. Зафиксировано.

### ТП, логистика, юристы, маркетинг, магазин, сервис

- **Понял:** Руты = реализация ТП по точкам за прошлый месяц; на карте выделение района и экспорт рутов. Логистика = дни отсутствия товара по городам + причины + расходы. Юристы = Excel-реестры или импорт своего реестра. Маркетинг = боты + Excel + дашборды. Магазин = по аналогии с городом/ТП, физлица и Kaspi. Сервис = реестр звонков, обратная связь, запчасти из 1С. Зафиксировано.
