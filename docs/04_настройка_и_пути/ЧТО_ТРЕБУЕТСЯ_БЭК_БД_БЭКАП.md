# Всё что требуется: бэкенд, БД, суббазы, бэкап и окружение

Один документ со списком всего необходимого для запуска и поддержки проекта: бэкенд, базы данных, папки-источники, бэкапы, переменные окружения и внешние сервисы.

---

## 1. Бэкенд (Backend)

| Требование | Описание |
|------------|----------|
| **Язык** | Python 3.10+ (проверено на 3.12). |
| **Каталог** | `backend/` в корне проекта. |
| **Зависимости** | Устанавливаются из `backend/requirements.txt`: FastAPI, uvicorn, SQLAlchemy, openpyxl, pandas, google-auth, google-api-python-client, python-jose, bcrypt, python-dotenv, aiofiles, email-validator и др. |
| **Установка** | `cd backend` → `python -m venv venv` → активация venv → `pip install -r requirements.txt`. |
| **Запуск** | `uvicorn app.main:app --reload --host 0.0.0.0 --port 8000` (из папки `backend` с активным venv). |
| **Инициализация БД** | Один раз: `python -m scripts.init_db` (создаёт таблицы и учётную запись admin@medhouse.kz / admin123). |

Бэкенд отдаёт API по адресу `http://127.0.0.1:8000`; документация Swagger — `http://127.0.0.1:8000/docs`.

## 2. База данных (основная БД)

| Требование | Описание |
|------------|----------|
| **СУБД** | **Сейчас:** SQLite. Файл БД по умолчанию: `backend/medhouse.db` (путь задаётся через `DATABASE_URL`). |
| **Для прода (рекомендация)** | PostgreSQL при 100+ пользователях. В `.env` задать `DATABASE_URL=postgresql://user:password@host:5432/dbname`. |
| **Миграции** | Отдельных миграций (Alembic) пока нет; схема создаётся при старте из `app/models.py` (`Base.metadata.create_all`). При смене СУБД или добавлении таблиц при необходимости ввести Alembic. |

**Таблицы в БД (модели в `app/models.py`):**

| Таблица | Назначение |
|---------|------------|
| `users` | Пользователи (email, пароль, роль: admin/moderator/operator). |
| `companies` | Компании (Медхаус, Свисс Энерджи). |
| `report_templates` | Шаблоны отчётов для скачивания. |
| `report_uploads` | Загруженные отчёты (имя файла, путь, период, компания). |
| `analysis_datasets` | Метаданные наборов для аналитики (Анализ 2 и др.). |
| `mailing_batches` | Партии рассылки из 1С (загруженный файл «Выгрузка XLSX»). |
| `mailing_rows` | Строки рассылки (период, организация, контрагент, номенклатура, сумма, profit, raw_data). |
| `google_sheets` | Реестр обнаруженных Google Таблиц (spreadsheet_id, название, папка). |
| `google_sheet_snapshots` | Снимки листов таблиц (данные в JSON). |

## 3. Суббазы и папки-источники (данные вне основной БД)

Это не отдельные СУБД, а каталоги и файлы, от которых зависит работа системы.

| Путь / переменная | Назначение | Обязательность |
|-------------------|------------|----------------|
| **Папка «база для данных»** (`DATA_BASE_DIR`) | Локальные файлы-источники: ОТЧЕТ МХК 2026.xlsx, Номенклатура.xlsx, Контрагент данные.xlsx, Адреса (2).xlsx; при наличии — .epf (1С), .pbix (Power BI). Путь по умолчанию: `<корень проекта>/база для данных`. Переопределение: `DATA_BASE_PATH` в `.env`. | Обязательно для загрузки справочников и ПНЛ; папка должна существовать. |
| **Папка «Анализ 2»** (`ANALIZ2_DIR`) | Годовые файлы аналитики (2020–2025, логистика). По умолчанию: `<корень проекта>/Анализ 2`. Переопределение: `ANALIZ2_PATH` в `.env`. | Нужна для дашборда аналитики. |
| **backend/uploads/** | Каталог загруженных пользователями отчётов (XLSX/XLS/CSV). Создаётся автоматически при старте. | Обязательно для раздела «Импорт отчётов». |
| **backend/templates/** | Файлы шаблонов отчётов (для скачивания). Создаётся при `init_db`. | Для раздела «Шаблоны». |

Проверка наличия и содержимого «база для данных» и Google Drive: `cd backend && python -m scripts.check_paths`.

## 4. Бэкап

| Что бэкапить | Частота (рекомендация) | Куда сохранять |
|--------------|------------------------|----------------|
| **Файл БД** `backend/medhouse.db` | Ежедневно или при каждом значимом изменении (загрузка рассылки, массовый импорт). | Отдельный диск или облако (например, копия в папку `backups/` с датой в имени: `medhouse_YYYY-MM-DD.db`). |
| **Папка backend/uploads/** | По мере появления важных загрузок или раз в день. | Копия в то же хранилище, что и бэкап БД. |
| **Папка «база для данных»** | По мере обновления файлов (ОТЧЕТ МХК, Номенклатура, Контрагент, Адреса). | Резервная копия в облако или на другой диск. |
| **Файл backend/.env** | При изменении переменных (секреты, ключи, пути). | Хранить в безопасном месте; не коммитить в Git. Копию .env можно положить в то же хранилище бэкапов с ограниченным доступом. |

Автоматизация: по ТЗ бэкап действий пользователей — каждые 3 часа; для файловой системы и БД достаточно cron/task scheduler (скрипт копирования `medhouse.db` и при необходимости `uploads/` и «база для данных»). Пример команды копирования БД (Windows):  
`copy backend\medhouse.db backups\medhouse_%date%.db` (или аналог с датой в имени).

## 5. Переменные окружения (полный список)

Файл **`backend/.env`** (скопировать из `backend/.env.example` и заполнить).

| Переменная | Обязательность | Описание |
|------------|----------------|----------|
| **SECRET_KEY** | Обязательно в продакшене | Секрет для JWT. Сгенерировать: `openssl rand -hex 32`. |
| **DATABASE_URL** | Обязательно | Подключение к БД. По умолчанию: `sqlite:///./medhouse.db`. Для PostgreSQL: `postgresql://user:password@host:5432/dbname`. **Supabase:** взять URI в панели проекта → Settings → Database → Connection string (URI). |
| **SUPABASE_URL**, **SUPABASE_ANON_KEY** | Опционально | Для прод с Supabase: из вкладки API Keys (при необходимости для клиентского доступа). Service Role Key — в API settings. |
| **LOG_LEVEL** | Опционально | Уровень логирования: DEBUG, INFO, WARNING, ERROR (по умолчанию INFO). |
| **LOG_FILE** | Опционально | Путь к файлу логов (относительно backend/ или абсолютный). Если не задан — только stderr. |
| **CORS_ORIGINS** | Опционально | В проде: разрешённые домены через запятую (например URL фронта на Vercel). Пусто — разрешены все (`*`). |
| **MEDHOUSE_API_KEY_1**, **_2**, **_3** | Опционально | API-ключи для доступа из Postman/внешних систем (полный доступ как admin). Сгенерировать: `python -m scripts.generate_api_key`. |
| **ALGORITHM** | Опционально | Алгоритм JWT, по умолчанию HS256. |
| **ACCESS_TOKEN_EXPIRE_MINUTES** | Опционально | Время жизни токена (минуты), по умолчанию 60. |
| **COMPANY_MEDHOUSE**, **COMPANY_SWISS** | Опционально | Названия компаний (по умолчанию Медхаус, Свисс Энерджи). |
| **DATA_BASE_PATH** | Опционально | Полный путь к папке «база для данных», если она не в корне проекта. |
| **ANALIZ2_PATH** | Опционально | Путь к папке «Анализ 2», если не в корне. |
| **GOOGLE_DRIVE_CREDENTIALS_FILE** | Для Google Drive (OAuth) | Имя файла учётных данных OAuth, по умолчанию `credentials.json` (положить в `backend/`). |
| **GOOGLE_DRIVE_TOKEN_FILE** | Для Google Drive (OAuth) | Имя файла токена, по умолчанию `token.json`. |
| **GOOGLE_DRIVE_SERVICE_ACCOUNT_FILE** | Для Google Drive (без браузера) | Имя файла ключа сервисного аккаунта, по умолчанию `service_account_pharmacy_bot.json` (положить в `backend/`). |
| **GOOGLE_DRIVE_PNL_FOLDER_ID** | Опционально | ID папки ПНЛ в Drive (по умолчанию задан). |
| **GOOGLE_DRIVE_EVGENIJ_FOLDER_ID** | Опционально | ID папки с форматом бюджета (…Евгений). |
| **GOOGLE_DRIVE_ALMAS_FOLDER_ID** | Опционально | ID папки с форматом бюджета (…Алмас). |
| **GOOGLE_MX_SPREADSHEET_ID** | Опционально | ID книги МХ формат (по умолчанию задан). |
| **GOOGLE_ZAJAVKA_FARMACEVTOV_SPREADSHEET_ID** | Опционально | ID таблицы «Заявка фармацевты» (по умолчанию задан). |
| **GOOGLE_ZAJAVKA_SHEET_GID** | Опционально | Номер листа (gid) в заявке фармацевты. |

## 6. Google Drive / Google API

| Требование | Описание |
|------------|----------|
| **Проект Google Cloud** | В проекте используется проект **pharmacy-bot-487900**. Нужны включённые API: Google Drive API, Google Sheets API. |
| **Доступ к Drive** | Один из вариантов: (1) **Сервисный аккаунт** — скачать JSON-ключ, переименовать в `service_account_pharmacy_bot.json`, положить в `backend/`. Выдать папкам/таблицам в Google доступ на чтение (или редактирование) для email сервисного аккаунта `pharmacy-bot@pharmacy-bot-487900.iam.gserviceaccount.com`. (2) **OAuth** — положить `credentials.json` в `backend/`, при первом запросе к Drive войти в браузере и выдать доступ. |
| **Что открыть для сервисного аккаунта** | Папка ПНЛ (ID в конфиге), при необходимости — папки …Евгений и …Алмас; таблица МХ формат; таблица «Заявка фармацевты». |

Подробнее: **GOOGLE_DRIVE.md** в корне проекта.

## 7. 1С

| Требование | Описание |
|------------|----------|
| **Прямое подключение к 1С** | Не используется. База 1С (файлы .1CD, .dt) читается только через 1С:Предприятие. |
| **Обмен данными** | Через выгрузки: Excel/CSV из 1С (отчёты по ОСВ 1010, 1030, 1210, остатки, рассылка и т.д.) загружаются вручную в папку «база для данных», в раздел «Импорт» на сайте или подставляются в Google-таблицы. В перспективе — HTTP API 1С, если появится. |

Расположение базы 1С и связь с проектом: **1C_БАЗА.md** в корне проекта.

## 8. Фронтенд (кратко)

| Требование | Описание |
|------------|----------|
| **Стек** | Node.js (LTS), npm. React, Vite. Каталог: `frontend/`. |
| **Установка** | `cd frontend` → `npm install`. |
| **Запуск** | `npm run dev` (по умолчанию http://localhost:3030; API проксируется на backend). |
| **Сборка** | `npm run build` — результат в `frontend/dist/`. |

Для продакшена собранный фронт раздаётся через Nginx или другой веб-сервер; запросы к API перенаправляются на бэкенд. Вариант **Vercel + Railway**: фронт на Vercel, бэкенд на Railway; в Vercel задаётся переменная `VITE_API_BASE` = URL бэкенда (см. **DEPLOY.md**).

## 9. Опционально / на будущее

| Компонент | Когда нужен |
|-----------|-------------|
| **PostgreSQL** | При росте нагрузки и 100+ пользователях вместо SQLite. |
| **Redis** | Кэш сессий и отчётов, очереди задач при масштабировании. |
| **Отдельное файловое хранилище (S3 и т.п.)** | Если загрузок много и хранить их на том же сервере нежелательно. |
| **Alembic** | Миграции схемы БД при добавлении/изменении таблиц в команде. |
| **Обратная связь с 1С по API** | Если в 1С появится HTTP-сервис для выгрузки данных — описать эндпоинты и добавить в конфиг. |

## 10. Чек-лист «Всё на месте»

- [ ] Python 3.10+ установлен; в `backend` создан venv и установлены зависимости из `requirements.txt`.
- [ ] Файл `backend/.env` создан из `.env.example`, заданы минимум `SECRET_KEY` и при необходимости `DATABASE_URL`.
- [ ] Выполнен `python -m scripts.init_db` в папке `backend`.
- [ ] Папка «база для данных» существует и в ней есть ожидаемые файлы (проверка: `python -m scripts.check_paths`).
- [ ] При необходимости: в `backend/` положен `service_account_pharmacy_bot.json` и папкам/таблицам в Google выдан доступ сервисному аккаунту.
- [ ] Регулярный бэкап `medhouse.db` и при необходимости `uploads/` и «база для данных» настроен (cron или аналог).

После выполнения пунктов выше бэкенд можно запускать (`uvicorn app.main:app ...`), а фронтенд — отдельно (`npm run dev` в `frontend/`).
